## 1. 제품 개요

| 항목 | 내용 |
|------|------|
| 제품명 | TodoList |
| 제품 유형 | 개인용 할일 관리 반응형 웹 애플리케이션 |
| 목적 | 카테고리 및 마감일 기반의 체계적인 개인 할일 관리 |
| 플랫폼 | 반응형 웹 (모바일 / 데스크탑) |
| 목표 KPI | 할일 등록 후 완료까지 평균 리드타임 감소, 기한 초과 할일 발생률 20% 감소 |

---

## 2. 문제 정의

- 카테고리 분류를 지원하는 적절한 할일 관리 앱의 부재
- 할일의 마감일(종료일)을 명확하게 표현하고 상태별로 구분하는 기능의 부재
- 개인화된 카테고리를 자유롭게 구성할 수 없는 기존 앱의 한계

## 4. 범위 정의 (Scope)

### 4.1 In Scope (구현 대상)

| 영역 | 기능 |
|------|------|
| 사용자 정보 관리 | 회원가입, 로그인, 비밀번호 재설정, 회원 탈퇴 |
| 할일 관리 | 할일 CRUD, 완료/취소 상태 전환, 마감일 지정, 상태별 시각 구분 |
| 카테고리 관리 | 카테고리 CRUD, 카테고리별 필터링 |

### 4.2 Out of Scope (구현 제외)

- 팀 공유 / 협업 기능
- 푸시 알림 / 이메일 알림
- 소셜 로그인 (Google, Kakao 등)
- 파일 첨부
- 반복 일정 (Recurring Todo)
- 다크 모드 → **Phase 2**
- 다국어 지원 (i18n) → **Phase 2**

## 6. 기능 요구사항

> 도메인 정의서 v1.1 기반 (docs/1-domain-definition.md)

### 6.1 인증 및 계정 (AUTH)

| ID | 요구사항 | 우선순위 |
|---|---|---|
| FR-AUTH-01 | 이메일/비밀번호 기반 회원가입 및 로그인 | Must |
| FR-AUTH-02 | 비인증 사용자는 로그인/회원가입 페이지 외 접근 불가 (리다이렉트) | Must |
| FR-AUTH-03 | 비밀번호 찾기 및 재설정 (이메일 인증 기반) | Should |
| FR-AUTH-04 | 회원 탈퇴 (본인 확인 후 Hard Delete) | Should |

**인증 방식**: JWT (알고리즘: HS-512)
- Access Token 발급 후 Authorization 헤더(Bearer)로 전달
- 토큰 만료 시 재로그인 유도

### 6.2 할일 관리 (TODO)

| ID | 요구사항 | 우선순위 |
|---|---|---|
| FR-TODO-01 | 할일 생성 · 조회 · 수정 · 삭제 (CRUD) | Must |
| FR-TODO-02 | 할일 작성 시 카테고리 및 종료일 지정 (선택 항목) | Must |
| FR-TODO-03 | 할일 완료 처리 및 완료 취소 (상태 전환) | Must |

### 6.3 카테고리 관리 (CAT)

| ID | 요구사항 | 우선순위 |
|---|---|---|
| FR-CAT-01 | 카테고리 생성 · 수정 · 삭제 | Must |
| FR-CAT-02 | 카테고리별 할일 필터링 및 목록 조회 | Must |

### 6.4 대시보드 및 표시 (VIEW)

| ID | 요구사항 | 우선순위 |
|---|---|---|
| FR-VIEW-01 | 완료 / 기한 초과 / 진행 중 할일의 시각적 구분 표시 | Must |
| FR-VIEW-02 | 기한 임박 할일(24시간 이내)에 대한 강조 표시 | Must |

## 7. 비기능 요구사항 (NFR)

| ID | 분류 | 요구사항 |
|---|---|---|
| NFR-SEC-01 | 보안 | 비밀번호는 bcrypt (cost factor 12 이상) 로 해시하여 저장 |
| NFR-SEC-02 | 보안 | JWT는 HS-512 알고리즘 사용, 시크릿 키는 환경변수로 관리 |
| NFR-SEC-03 | 보안 | 모든 API는 인증 미들웨어를 통과해야 하며, 타인 데이터 접근 차단 |
| NFR-PER-01 | 성능 | 할일 목록 조회 API 응답 속도 200ms 이내 |
| NFR-PER-02 | 성능 | 프론트엔드 초기 로딩(LCP) 2.5초 이내 |
| NFR-USA-01 | 사용성 | 반응형 웹 디자인 — 모바일(360px~) / 데스크탑(1280px~) 모두 지원 |
| NFR-REL-01 | 안정성 | Phase 1 기간 중 핵심 기능(CRUD, 인증) 오류율 1% 미만 유지 |

## 8. 기술 스택 및 제약사항

### 8.1 아키텍처

```
[Browser] ──HTTP/REST──▶ [Express 5 API Server] ──SQL──▶ [PostgreSQL]
```

3-Tier 구성: Frontend / Backend API / Database

### 8.2 기술 스택

| 계층 | 기술 | 비고 |
|------|------|------|
| **Frontend** | React 19 | JavaScript (TypeScript 미사용) |
| | TanStack Query | 서버 상태 관리 (캐싱, 동기화) |
| | Zustand | 클라이언트 전역 상태 관리 |
| | Tailwind CSS | 유틸리티 기반 스타일링 |
| **Backend** | Node.js 24 | JavaScript (TypeScript 미사용) |
| | Express 5 | REST API 프레임워크 |
| | pg (node-postgres) | PostgreSQL 직접 쿼리 |
| | jsonwebtoken | JWT 발급/검증 (HS-512) |
| | bcrypt | 비밀번호 해싱 |
| **Database** | PostgreSQL | 관계형 DB |

### 8.3 제약사항

- TypeScript 미사용 (프론트/백엔드 모두 순수 JavaScript)
- ORM 미사용 — `pg` 라이브러리로 직접 SQL 작성
- 외부 알림 서비스 미사용 (Phase 1)

## 9. 디자인 방향

- **스타일링**: Tailwind CSS 유틸리티 클래스 기반
- **반응형**: 모바일 우선(Mobile First) 설계
- **상태 색상 가이드**

  | 상태 | 색상 방향 |
  |------|-----------|
  | 진행 중 (In Progress) | 기본 (흰/회색 계열) |
  | 기한 임박 (24h 이내) | 주황/노랑 강조 |
  | 기한 초과 (Overdue) | 빨강 계열 |
  | 완료 (Completed) | 초록 계열 + 취소선 |

- **다크 모드**: Phase 2에서 Tailwind `dark:` 클래스 기반으로 구현
- **다국어(i18n)**: Phase 2에서 `react-i18next` 도입 예정

## 10. 도메인 규칙 요약

| ID | 핵심 규칙 |

| 지표 | 목표값 | 측정 방법 |
|------|--------|-----------|
