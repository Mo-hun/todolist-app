### SCN-AUTH-01: 신규 회원가입

- **시나리오 ID**: SCN-AUTH-01
- **시나리오 제목**: 신규 회원가입
- **관련 페르소나**: 페르소나 A — 김철수 (27세, IT 스타트업 마케터)

**전제 조건 (Pre-condition)**
- 사용자가 앱에 처음 접속한 상태이다.
- 사용자는 아직 계정이 없다.
- 회원가입 페이지(`/register`)가 정상 로드된 상태이다.

**트리거**
업무 할일과 사이드 프로젝트 태스크를 한 곳에서 관리하고 싶었던 김철수가 동료 추천으로 TodoList 앱을 알게 되어 회원가입 버튼을 클릭한다.

**정상 흐름 (Main Flow)**

1. 김철수가 회원가입 페이지에 접속한다.
2. 이메일 입력 필드에 자신의 이메일 주소(`chulsoo.kim@startup.io`)를 입력한다.
3. 비밀번호 입력 필드에 8자 이상의 비밀번호를 입력하고, 비밀번호 확인 필드에 동일한 값을 입력한다.
4. 회원가입 버튼을 클릭한다.
5. 시스템이 이메일 중복 여부를 확인한다 (BR-AUTH-02).
6. 중복이 없으면 시스템이 비밀번호를 bcrypt로 해싱하여 DB에 저장한다.
7. 회원가입이 완료되고 로그인 페이지로 자동 리다이렉트된다.
8. 김철수는 가입된 계정으로 즉시 로그인할 수 있다.

**예외/대안 흐름 (Alternative Flow)**

| 단계 | 예외 상황 | 처리 방법 |
|------|-----------|-----------|
| 5 | 동일한 이메일로 이미 가입된 계정이 존재한다. | 시스템이 "이미 사용 중인 이메일입니다." 오류 메시지를 표시하고 이메일 필드에 포커스를 유지한다. |
| 3 | 비밀번호와 비밀번호 확인 값이 일치하지 않는다. | 시스템이 "비밀번호가 일치하지 않습니다." 오류 메시지를 표시한다. |
| 2 | 이메일 형식이 올바르지 않다. | 시스템이 "올바른 이메일 형식을 입력해주세요." 오류 메시지를 표시한다. |
| 3 | 비밀번호가 최소 길이 조건을 충족하지 않는다. | 시스템이 비밀번호 정책 안내 메시지를 표시한다. |

**사후 조건 (Post-condition)**
- 신규 사용자 레코드가 DB에 생성된다.
- 비밀번호는 bcrypt 해시값으로 저장되며 평문은 저장되지 않는다.
- 사용자는 로그인 페이지로 이동된다.

**관련 FR ID**: FR-AUTH-01 / 관련 BR: BR-AUTH-02

### SCN-AUTH-02: 로그인

- **시나리오 ID**: SCN-AUTH-02
- **시나리오 제목**: 로그인
- **관련 페르소나**: 페르소나 B — 박소영 (34세, 전업주부)

**전제 조건 (Pre-condition)**
- 박소영은 이미 가입된 계정(`soyoung.park@home.com`)이 있다.
- 로그인 페이지(`/login`)가 정상 로드된 상태이다.

**트리거**
아이 유치원 준비물 목록과 장보기 리스트를 확인하려고 아침에 스마트폰으로 앱을 열었으나 로그인이 만료된 상태이다.

**정상 흐름 (Main Flow)**

1. 박소영이 로그인 페이지에 접속한다.
2. 이메일 입력 필드에 `soyoung.park@home.com`을 입력한다.
3. 비밀번호 입력 필드에 비밀번호를 입력한다.
4. 로그인 버튼을 클릭한다.
5. 시스템이 이메일과 비밀번호를 검증한다.
6. 인증 성공 시 시스템이 JWT Access Token(HS-512)을 발급한다.
7. 발급된 토큰이 클라이언트에 저장된다.
8. 할일 목록 메인 페이지로 리다이렉트된다.

**예외/대안 흐름 (Alternative Flow)**

| 단계 | 예외 상황 | 처리 방법 |
|------|-----------|-----------|
| 5 | 이메일 또는 비밀번호가 일치하지 않는다. | 시스템이 "이메일 또는 비밀번호가 올바르지 않습니다." 오류 메시지를 표시한다. (보안상 어느 쪽이 틀렸는지 구분하지 않는다.) |
| 2 | 이메일 형식이 올바르지 않다. | 시스템이 필드 하단에 유효성 오류 메시지를 표시한다. |
| — | 비인증 상태에서 메인 페이지에 직접 접근한다. | 시스템이 로그인 페이지로 자동 리다이렉트한다 (BR-AUTH-01). |

**사후 조건 (Post-condition)**
- 유효한 JWT Access Token이 클라이언트에 저장된다.
- 박소영은 자신의 할일 목록 페이지에 접근할 수 있다.
- 이후 모든 API 요청에 `Authorization: Bearer {token}` 헤더가 포함된다.

**관련 FR ID**: FR-AUTH-01, FR-AUTH-02 / 관련 BR: BR-AUTH-01

### SCN-AUTH-03: 비밀번호 재설정

- **시나리오 ID**: SCN-AUTH-03
- **시나리오 제목**: 비밀번호 재설정
- **관련 페르소나**: 페르소나 A — 김철수 (27세, IT 스타트업 마케터)

**전제 조건 (Pre-condition)**
- 김철수는 가입된 계정(`chulsoo.kim@startup.io`)이 있다.
- 비밀번호를 기억하지 못하는 상태이다.
- 가입에 사용한 이메일에 접근할 수 있다.

**트리거**
업무 회의 중 급하게 할일을 확인하려고 앱을 열었으나 비밀번호가 기억나지 않아 로그인 페이지 하단의 "비밀번호를 잊으셨나요?" 링크를 클릭한다.

**정상 흐름 (Main Flow)**

1. 김철수가 로그인 페이지에서 "비밀번호를 잊으셨나요?" 링크를 클릭한다.
2. 비밀번호 재설정 요청 페이지가 표시된다.
3. 이메일 입력 필드에 `chulsoo.kim@startup.io`를 입력하고 전송 버튼을 클릭한다.
4. 시스템이 해당 이메일로 가입된 계정이 존재하는지 확인한다.
5. 계정이 확인되면 시스템이 비밀번호 재설정 링크를 이메일로 발송한다.
6. 화면에 "입력하신 이메일로 재설정 링크를 발송했습니다." 안내 메시지가 표시된다.
7. 김철수가 이메일을 열어 재설정 링크를 클릭한다.
8. 새 비밀번호 입력 페이지가 표시된다.
9. 새 비밀번호와 비밀번호 확인을 입력하고 변경 버튼을 클릭한다.
10. 시스템이 새 비밀번호를 bcrypt로 해싱하여 DB를 업데이트한다.
11. "비밀번호가 변경되었습니다." 메시지와 함께 로그인 페이지로 이동한다.

**예외/대안 흐름 (Alternative Flow)**

| 단계 | 예외 상황 | 처리 방법 |
|------|-----------|-----------|
| 4 | 입력한 이메일로 가입된 계정이 없다. | 보안상 동일하게 "재설정 링크를 발송했습니다." 메시지를 표시하여 계정 존재 여부를 노출하지 않는다. |
| 7 | 재설정 링크가 만료되었다. | 시스템이 "링크가 만료되었습니다. 다시 요청해주세요." 오류 페이지를 표시한다. |
| 9 | 새 비밀번호와 확인 값이 일치하지 않는다. | 시스템이 오류 메시지를 표시하고 입력 필드를 초기화한다. |

**사후 조건 (Post-condition)**
- DB에 새로운 bcrypt 해시 비밀번호가 저장된다.
- 기존에 발급된 JWT 토큰은 만료 처리된다.
- 김철수는 새 비밀번호로 로그인할 수 있다.

**관련 FR ID**: FR-AUTH-03

### SCN-AUTH-04: 회원 탈퇴

- **시나리오 ID**: SCN-AUTH-04
- **시나리오 제목**: 회원 탈퇴
- **관련 페르소나**: 페르소나 B — 박소영 (34세, 전업주부)

**전제 조건 (Pre-condition)**
- 박소영은 로그인된 상태이다.
- 박소영의 계정에는 기존에 등록한 할일과 카테고리 데이터가 존재한다.
- 계정 설정 페이지에 접근한 상태이다.

**트리거**
앱을 더 이상 사용하지 않기로 결정한 박소영이 계정 설정 메뉴에서 "회원 탈퇴" 버튼을 클릭한다.

**정상 흐름 (Main Flow)**

1. 박소영이 계정 설정 페이지에서 "회원 탈퇴" 버튼을 클릭한다.
2. 시스템이 탈퇴 확인 다이얼로그를 표시한다. ("탈퇴 시 모든 데이터가 즉시 삭제되며 복구할 수 없습니다.")
3. 본인 확인을 위해 현재 비밀번호 입력 필드가 제공된다.
4. 박소영이 현재 비밀번호를 입력하고 "탈퇴 확인" 버튼을 클릭한다.
5. 시스템이 비밀번호를 검증하여 본인임을 확인한다.
6. 시스템이 해당 사용자의 모든 데이터(할일, 카테고리, 계정 정보)를 Hard Delete한다 (BR-DATA-05).
7. JWT 토큰이 무효화된다.
8. 로그인 페이지로 리다이렉트되며 "탈퇴가 완료되었습니다." 메시지가 표시된다.

**예외/대안 흐름 (Alternative Flow)**

| 단계 | 예외 상황 | 처리 방법 |
|------|-----------|-----------|
| 5 | 입력한 비밀번호가 일치하지 않는다. | 시스템이 "비밀번호가 올바르지 않습니다." 오류 메시지를 표시하고 탈퇴를 중단한다. |
| 2 | 박소영이 확인 다이얼로그에서 취소 버튼을 클릭한다. | 다이얼로그가 닫히고 계정 설정 페이지로 복귀한다. 데이터는 변경되지 않는다. |

**사후 조건 (Post-condition)**
- 사용자 계정, 할일, 카테고리 레코드가 DB에서 완전히 삭제된다 (Hard Delete, 복구 불가).
- 탈퇴한 이메일로 재가입 시 신규 사용자로 처리된다.
- 박소영은 앱에 접근할 수 없으며 로그인 페이지로 이동된다.

**관련 FR ID**: FR-AUTH-04 / 관련 BR: BR-DATA-05

### SCN-CAT-01: 카테고리 생성

- **시나리오 ID**: SCN-CAT-01
- **시나리오 제목**: 카테고리 생성
- **관련 페르소나**: 페르소나 A — 김철수 (27세, IT 스타트업 마케터)

**전제 조건 (Pre-condition)**
- 김철수는 로그인된 상태이다.

**트리거**
새로운 사이드 프로젝트가 시작되어 기존 "업무" 카테고리와 별도로 "사이드 프로젝트" 카테고리를 생성하려는 김철수가 "카테고리 추가" 버튼을 클릭한다.

3. "사이드 프로젝트"를 입력하고 저장 버튼을 클릭한다.
4. 시스템이 해당 사용자의 카테고리로 DB에 저장한다.
5. 카테고리 목록에 "사이드 프로젝트"가 즉시 표시된다.

**예외/대안 흐름 (Alternative Flow)**

| 단계 | 예외 상황 | 처리 방법 |
|------|-----------|-----------|
| 3 | 카테고리 이름을 입력하지 않고 저장을 시도한다. | 시스템이 "카테고리 이름을 입력해주세요." 오류 메시지를 표시한다. |
| 3 | 동일한 이름의 카테고리가 이미 존재한다. | 시스템이 "동일한 이름의 카테고리가 이미 있습니다." 오류 메시지를 표시한다. |
| — | 네트워크 오류 등으로 저장에 실패한다. | 시스템이 오류 메시지를 표시하고 입력 내용을 유지한다. |

**사후 조건 (Post-condition)**
- 새 카테고리 레코드가 해당 사용자(`user_id`)와 연결되어 DB에 저장된다.
- 카테고리 목록에 즉시 반영되어 할일 생성/수정 시 선택 가능한 상태가 된다.

**관련 FR ID**: FR-CAT-01 / 관련 BR: BR-AUTH-03

### SCN-CAT-02: 카테고리 수정

### SCN-TODO-06: 기한 초과 할일 확인

- **시나리오 ID**: SCN-TODO-06
- **시나리오 제목**: 기한 초과 할일 확인
- **관련 페르소나**: 페르소나 A — 김철수 (27세, IT 스타트업 마케터)

**전제 조건 (Pre-condition)**
- 김철수는 로그인된 상태이다.
- 종료일이 이미 경과한 미완료 할일들이 목록에 존재한다.
- 종료일이 없는 할일도 목록에 존재한다.
- 완료 처리된 할일 중 종료일이 경과한 항목도 존재한다.
- 할일 목록 페이지에 접근한 상태이다.

**트리거**
주말을 보내고 월요일 아침에 출근한 김철수가 밀린 업무 현황을 파악하기 위해 앱 할일 목록을 열었을 때 빨간색으로 강조된 항목들을 발견한다.

**정상 흐름 (Main Flow)**

1. 김철수가 할일 목록 메인 페이지에 접속한다.
2. 시스템이 현재 일시와 각 할일의 종료일을 비교한다 (BR-DATA-01).
3. 기한 초과 할일 판별 조건을 적용한다:
   - `due_date < 현재 일시` AND `is_completed = false` → 기한 초과 (Overdue)
   - `due_date`가 null인 항목 → 기한 초과 판별 제외 (BR-DATA-04)
   - `is_completed = true`인 항목 → 기한 초과 판별 제외 (BR-DATA-02)
4. 기한 초과 할일들이 빨강 계열 색상으로 강조 표시된다.
   - 예: "지난주 발행 예정 블로그 포스팅" — 빨강 강조 (기한 초과)
   - 예: "개인 운동 루틴 기록" — 기본 색상 (종료일 없음, 판별 제외)
   - 예: "완료된 기획서 제출" — 초록 + 취소선 (완료, 기한 초과 표시 안 함)
5. 김철수가 기한 초과 할일을 즉시 식별하고 우선순위를 재조정한다.

**예외/대안 흐름 (Alternative Flow)**

| 단계 | 예외 상황 | 처리 방법 |
|------|-----------|-----------|
| 2 | 기한 초과 할일이 없는 경우. | 빨강 강조 항목 없이 목록이 정상 표시된다. |
| — | 자정이 지나면서 새로운 기한 초과 항목이 발생한다. | 페이지 재접속 또는 갱신 시 업데이트된 상태가 자동 반영된다. |

**사후 조건 (Post-condition)**
- 기한 초과 할일이 빨강 계열로 시각 구분되어 표시된다.
- 종료일 없는 할일과 완료된 할일은 기한 초과로 표시되지 않는다.
- 김철수는 즉시 처리가 필요한 항목을 한눈에 파악할 수 있다.

**관련 FR ID**: FR-VIEW-01, FR-VIEW-02 / 관련 BR: BR-DATA-01, BR-DATA-02, BR-DATA-04
